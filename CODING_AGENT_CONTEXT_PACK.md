# CODING_AGENT_CONTEXT_PACK.md

# Coding agent context pack: how the hub talks to agents

A context pack is a structured prompt the hub generates when invoking a coding agent (Claude Code, Copilot, an internal agent) against a specific repo or task. It is the only thing the agent should receive — never the full bundle, never the operating spec, never another client's artifacts.

The pack is generated by the hub's `generate-context-pack` script, which reads the target repo's manifest, the relevant playbook docs, and the agent's stated task, then assembles a scoped pack with strict allow/deny lists.

## The template

```markdown
# Coding Agent Context Pack

## Generated
2026-05-01T14:23:00Z by hub generate-context-pack v1

## Target repo
example-corp-os (sensitivity_tier: 2, client_id: example-corp)

## Current phase
Phase 0.5: artifact harvest from first paid audit.

## Goal
Implement the intake triage automation specified in
requirements/intake-triage-spec.md. Do not exceed the appetite
defined in the spec.

## Must read
- engineering-playbook/HANDBOOK.md (raw URL)
- engineering-playbook/templates/RUNBOOK.md (raw URL)
- engineering-playbook/templates/DECISION_LOG.md (raw URL)
- this repo's ARCHITECTURE.md
- this repo's requirements/intake-triage-spec.md

## Must not read
- Any other client repo
- hub-prompts internals
- engineering-playbook/overlays/* unless explicitly named
- Any artifact with sensitivity_tier: 3 outside this engagement

## Must not build
- A CRM
- A billing dashboard
- A time tracker
- A client portal
- Vector search
- Full cockpit navigation
- Anything not in the spec

## Files to create
- automations/intake-triage.ts
- runbooks/intake-triage-runbook.md
- decisions/2026-05-01-intake-triage-design.md

## Acceptance criteria
- intake-triage.ts passes the test cases in
  requirements/intake-triage-spec.md#test-cases
- intake-triage-runbook.md follows the runbook template structure
- decision log entry is signed and dated, and references the
  spec by version
- no new dependencies added without entry in decisions/
- all created files include valid frontmatter per
  ARTIFACT_FRONTMATTER_SCHEMA.md
- sensitivity_tier is set explicitly on every artifact

## Out of scope
- UI changes
- Integration with systems not named in the spec
- Refactoring existing automations
- Production deployment (handoff packet covers this separately)

## Sensitivity rules
This is a tier 2 task inside a tier 3 client repo. The agent
may read tier 1 and tier 2 internal references and this repo's
own tier 2 and tier 3 artifacts. The agent may not read other
client repos, regardless of tier.

## Failure modes to flag, not work around
- Spec is ambiguous: stop, ask
- A required dependency is missing: stop, log
- Acceptance criteria conflict: stop, ask
- Scope appears to expand: stop, ask
```

## Generation rules

The hub builds the pack by:

1. Reading the target repo's `.repo.yml`
2. Reading the engagement's `clients.yml` entry
3. Pulling the named playbook templates as raw URLs
4. Computing the allow/deny lists from sensitivity tiers + `allowed_context_consumers`
5. Inserting the practitioner-authored "goal," "files to create," and "acceptance criteria" from the task definition

## Why this shape

The pack is opinionated about what *not* to do. Coding agents are good at producing plausible code; they are weakest at refusing scope expansion and respecting boundaries. The pack frontloads constraints because constraints are where agents fail silently. The "must not build" list is non-negotiable; the agent is required to flag and stop, not work around.

## What this is not

Not a Claude Code config file. Not a Copilot rules file. Not a system prompt. The pack is the *content* the agent receives; the delivery mechanism is whatever your agent tooling supports (a paste, a file in `.claude/`, a system prompt addition). The hub generates the pack; the practitioner delivers it.
